# HydroComplete Engine — Deep Audit Report (2026-07)

_Adversarially-verified correctness / robustness / security audit of `HydroComplete.Engine` + the Civil 3D plugin. 12 domain-expert finders; each finding double-verified (standard re-derivation + counterexample trace), then synthesized._

**Result:** 33 raised → 31 survived verification (30 CONFIRMED, 1 PLAUSIBLE). **23 fixed** on this branch; 7 deferred (design decision, external data, a tested-API re-baseline, or an uncompilable-here plugin change).

> ⚠️ **The xunit suite could not be run here (no .NET SDK).** Every applied fix is backed by two independent adversarial agents + a manual read against the cited standard, and `AuditRegressionTests.cs` locks several corrected values. **Run `dotnet test` before merging** — the hydrograph/runoff/routing fixes change outputs, so some tests that pinned the old behavior will need re-baselining.

## Executive summary

HydroComplete is NOT ready for Autodesk App Store submission. Across 31 adversarially-verified findings (30 after de-duplication; 30 CONFIRMED, 1 PLAUSIBLE), the audit found that nearly every core computational pathway an engineer would stamp can produce silently-wrong results. There are 5 critical and 6 high-severity correctness defects that corrupt numbers users trust: SCS runoff (curve-number relationship inverted, so pavement is treated as pervious), culvert HDS-5 outlet control (an SI friction constant used in US-customary code, ~33% low), bioretention drawdown/pollutant removal (underdrain flow off by a factor of 3600), Washington State compliance (a 91-inch design-storm decimal typo that guarantees a false pass and inflates WQV two orders of magnitude), and the flagship BMP cost-effectiveness ranking (two zero-SizingFactor BMPs always rank #1 at $0.00/lb). One entire conduit type — pipe-arch — throws an exception on essentially all realistic AISI/NCSPA sizes because the arc-radius sagitta constant is halved. Two reviewer-reachable failure modes (uncaught StackOverflow on a looped pipe topology; an infinite loop on a zero elevation step) can tear down the Civil 3D process, and a license-validation gap grants Pro on any reachable-but-denying server response.

A clear root-cause pattern runs through the worst findings: dimensional / unit-system errors (SI constants in ft/g=32.2 code, missing acre-inch→ft³ and per-second→per-hour conversions) and inverted or mis-specified physical relationships. These are exactly the class of bug that a numeric-value regression test would have caught, and the test suite (F31) only asserts loose magnitude bounds — never against published FHWA/NRCS constants — which is why they shipped. Recommendation: block submission until all critical and high correctness findings plus the crash/hang set are fixed AND locked with numeric regression tests validated against hand-computed reference values. Several fixes are one-line (WA typo F17, friction constant F3, cycle guard F6, step guard F24), so the critical path is tractable, but the hydrograph/runoff corrections change computed outputs and require re-baselining the suite.

## Deferred findings — precise fixes (apply with a test run)

### [F4] MEDIUM · `src/HydroComplete.Engine/CulvertHydraulics.cs:113-115` — Inlet-control unsubmerged form hard-codes Hc/D = 1.0 instead of the critical-head ratio

Hc/D is a function of discharge (Hc = yc + Vc^2/2g) and approaches 0 as Q -> 0. - **Suggested fix:** Compute Hc/D from critical depth and critical velocity head for the circular section at the given Q (Hc = dc + Q^2/(2 g A_c^2)) and use it in place of the literal 1.0, or at minimum let the unsubmerged branch's constant scale toward 0 at low flow rather than being pinned at 1.0.

### [F10] MEDIUM · `src/HydroComplete.Engine/SnyderUnitHydrograph.cs:78, 125-138` — Snyder triangular UH base time tb = 5·tp does not enclose 1 inch — hydrograph volume overstated ~49%

- **Why deferred:** Fix changes the public `BaseTimeHours` API which has existing tests; needs a coordinated re-baseline under `dotnet test`.
- **Detail:** Generate builds a triangle rising to qp at tp and receding to tb = BaseTimeHours(tp) = max(5·tp, 3) (line 78, 97, 125-138). Its enclosed volume is 0.5·qp·tb = 2.5·qp·tp. With qp = 640·Cp·A/tp (cfs, A in mi^2) this is 1600·Cp·A cfs·hr, whereas one inch of direct runoff over A mi^2 is 645.33·A cfs·hr. The ratio is 2.479·Cp — for the default Cp = 0.6 the UH encloses ~1.49 inches (≈49% too much volume), and for tp < 0.6 h the max(…,3) floor makes it worse. A unit hydrograph must, by definition, enclose exactly one unit of direct runoff, so every Snyder-based convolved hydrograph (and its reported 
- **Suggested fix:** Set the triangular base so volume = 1 inch: tb = 2·V_unit/qp where V_unit(cfs·hr) = 645.33·A_mi^2, i.e. tb = 2·645.33·A/(640·Cp·A/tp) = 2.017·tp/Cp; or normalize the generated ordinates so their trapezoidal integral equals the unit volume before returning.

- **Suggested fix:** Split the post-colon substring on ',' WITHOUT discarding empty entries (keep them as positional placeholders, only Trim each), then index values[ariColumnIndex] directly; treat an empty/blank cell at the requested column as 'no data for this return period' (return false) rather than pulling a neighboring column's value.

### [F31] MEDIUM · `tests/HydroComplete.Engine.Tests/CulvertHydraulicsTests.cs:` — CulvertHydraulics inlet/outlet-control HW is only asserted by loose magnitude bounds, never against FHWA HDS-5 constants

- **Why deferred:** Partially addressed — AuditRegressionTests.cs added; still recommend FHWA HDS-5 culvert reference-value assertions.
- **Detail:** CulvertHydraulicsTests.cs asserts only q>10 && q<25 (ManningFullFlow), HeadwaterFt>0 (50 cfs), and rating-curve monotonicity. No test pins a numeric headwater value, so the FHWA HDS-5 inlet-control constants in Headwater() (ku=0.0098, mu=2.0, ksu=-0.5, cs=0.0398, ys=0.67 at lines 107-114) and the outlet-control loss/critical-depth terms (frictionCoeff=19.63*n^2*L/R^(4/3) line 118; dc=0.467*(Q^2/(g*D^5))^0.1*D line 120-121) are unverified — any of these could be transposed and every test would still pass. Note also the unsubmerged inlet form uses a fixed '1.0' in place of the HDS-5 Hc/D critica
- **Suggested fix:** Add tests that verify HW/D against a hand-computed FHWA value: for the 24-in default culvert at a discharge that yields submerged inlet control, assert HeadwaterInletFt equals D*(cs*(Q/(A*D^0.5))^2 + ys - 0.5*S) to 2 decimals; assert Control switches from Inlet to Outlet at the expected discharge; and add an unsubmerged-flow case asserting the computed HW so the '1.0' vs Hc/D simplification is documented and locked.

### [F20] LOW · `src/HydroComplete.Engine/ProfileDxfWriter.cs:110-114` — DXF single-line TEXT label emits literal backslash-n instead of line breaks

- **Why deferred:** Cosmetic DXF label line-breaks; fix restructures TEXT->MTEXT and can't be visually verified here.
- **Detail:** The station label is built with C# escaped `\\n` (a literal backslash + 'n', two characters) and written as a group-code-1 value on a DXF TEXT entity via WriteText (lines 177-183). DXF TEXT/DTEXT is a single-line entity: it does not interpret `\n`, and backslash escape sequences are an MTEXT-only feature. AutoCAD renders the label verbatim as e.g. `MH-1\nSTA 0.0\nHGL 12.34` on one line rather than three lines. The label content (structure name, station, HGL) is correct, only the intended line breaking is broken, so this is cosmetic and does not corrupt any stamped numeric value.
- **Suggested fix:** Either emit one TEXT entity per line with a stacked Y (subtract textH*1.4 per line), or switch the label to an MTEXT (0/MTEXT) entity and use the MTEXT paragraph code `\P` instead of `\n`. Do not rely on `\n` in a TEXT entity.

### [F30] LOW · `src/HydroComplete.Engine/SsugroFetcher.cs:358-373` — No response-size cap before buffering remote body into memory (all three fetchers)

- **Why deferred:** LOW defense-in-depth; touches shared HttpClient construction in 3 fetchers.
- **Detail:** DownloadSdaAsync (SsugroFetcher.cs:372), Atlas14Fetcher.DownloadCsvAsync (Atlas14Fetcher.cs:202), and LicenseActivator.TryOnlineValidationAsync (LicenseActivator.cs:239) each call response.Content.ReadAsStringAsync() with no MaxResponseContentBufferSize and no Content-Length/size guard, buffering the entire response into a string. A malformed, redirected, or oversized response (the code even documents a 301-redirect chain at Atlas14Fetcher.cs:20-25) is read wholesale into memory before any shape validation. Impact is bounded because the endpoints are trusted government/first-party hosts over T
- **Suggested fix:** Set MaxResponseContentBufferSize on the shared HttpClients (e.g. a few MB — these CSV/JSON payloads are small) or check response.Content.Headers.ContentLength before ReadAsStringAsync, returning the existing cache/fallback path when exceeded.

## Roadmap

- **P0** — Fix the 5 critical and 6 high correctness defects that corrupt stamped results: WA storm typo (F17), C→CN inversion (F5/F22), bioretention ×3600 (F12), culvert friction constant (F3), pipe-arch radius crash (F1), BMP zero-cost ranking (F14), continuous-sim RNG overflow (F23), Clark UH cfs conversion (F7) and zero-histogram (F8), GVF supercritical sign (F2), Muskingum-Cunge timestep (F9).
  - _Each produces silently-wrong numbers an engineer stamps, or (F1) crashes an entire conduit type. Shipping any of these to the App Store risks a professional relying on incorrect hydrology/hydraulics — a liability and reputational blocker. Several are one-liners (F17, F3, F23)._
- **P0** — Add the two cheap, auto-fix-safe crash/hang guards a reviewer can trivially trigger: cycle guard in LongestPathFrom (F6, uncatchable StackOverflow tears down Civil 3D) and the elevStepFt<=0 guard in BuildStorageIndicationCurve (F24, infinite loop).
  - _An App Store reviewer feeding a malformed or edge-case model must not be able to crash or freeze the host CAD process; these are low-risk guard-only changes._
- **P0** — Lock every P0 fix with a numeric regression test against a hand-computed reference (HDS-5 HW, TR-55 CN anchors C≈0.95→CN~98, unit-volume checks for all three UH methods, a real 43×27 pipe-arch). Do not merge a correctness fix without its anchor test.
  - _The systemic cause of these bugs shipping is that tests assert only loose magnitude bounds (F31); numeric anchors are the only durable guard against regression and against the remaining transposition-class errors._
- **P1** — Resolve the medium correctness findings: WQ load double-count (F15), box wetted perimeter (F11), pond-chain tailwater double-subtraction (F13), inlet-control Hc/D (F4), HGL label key collision (F26), and the ~27-jurisdiction drawdown window (F18). Investigate the PLAUSIBLE Atlas14 column-shift (F29) and confirm/fix.

- **P1** — Close the license-validation gap (F28): treat 4xx and non-JSON/non-boolean server responses as denials, reserving the offline stub for true transport failures. Add the remaining input guards (F25 Years, F27 report write) and the fetcher response-size cap (F30).

- **P1** — Expand the test suite beyond magnitude bounds across the engine (F31): numeric anchors for SCS/CN, culvert control transition, hydrograph unit-volume and inter-method consistency, and a StateComplianceData table-validation test asserting sane storm/drawdown ranges for every jurisdiction.
