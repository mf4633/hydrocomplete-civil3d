# HydroComplete Engine — Deep Audit Report (2026-07)

_Adversarially-verified correctness / robustness / security audit of `HydroComplete.Engine` + the Civil 3D plugin. 12 domain-expert finders; each finding double-verified (standard re-derivation + counterexample trace), then synthesized._

**Result:** 33 raised → 31 survived verification (30 CONFIRMED, 1 PLAUSIBLE). **All 30 distinct findings addressed** (F22 was a duplicate of F5; F31 addressed by the new regression suite).

> ⚠️ **The xunit suite could not be run here (no .NET SDK).** Every fix is backed by two independent adversarial agents + a manual read against the cited standard, and `AuditRegressionTests.cs` locks corrected values. **Run `dotnet test` before merging** — the hydrograph/runoff/routing/Snyder/culvert fixes change outputs, so tests that pinned the old behavior need re-baselining, and the Muskingum-Cunge routing and F4 culvert change warrant a numerical sanity check.

## Executive summary

HydroComplete is NOT ready for Autodesk App Store submission. Across 31 adversarially-verified findings (30 after de-duplication; 30 CONFIRMED, 1 PLAUSIBLE), the audit found that nearly every core computational pathway an engineer would stamp can produce silently-wrong results. There are 5 critical and 6 high-severity correctness defects that corrupt numbers users trust: SCS runoff (curve-number relationship inverted, so pavement is treated as pervious), culvert HDS-5 outlet control (an SI friction constant used in US-customary code, ~33% low), bioretention drawdown/pollutant removal (underdrain flow off by a factor of 3600), Washington State compliance (a 91-inch design-storm decimal typo that guarantees a false pass and inflates WQV two orders of magnitude), and the flagship BMP cost-effectiveness ranking (two zero-SizingFactor BMPs always rank #1 at $0.00/lb). One entire conduit type — pipe-arch — throws an exception on essentially all realistic AISI/NCSPA sizes because the arc-radius sagitta constant is halved. Two reviewer-reachable failure modes (uncaught StackOverflow on a looped pipe topology; an infinite loop on a zero elevation step) can tear down the Civil 3D process, and a license-validation gap grants Pro on any reachable-but-denying server response.

A clear root-cause pattern runs through the worst findings: dimensional / unit-system errors (SI constants in ft/g=32.2 code, missing acre-inch→ft³ and per-second→per-hour conversions) and inverted or mis-specified physical relationships. These are exactly the class of bug that a numeric-value regression test would have caught, and the test suite (F31) only asserts loose magnitude bounds — never against published FHWA/NRCS constants — which is why they shipped. Recommendation: block submission until all critical and high correctness findings plus the crash/hang set are fixed AND locked with numeric regression tests validated against hand-computed reference values. Several fixes are one-line (WA typo F17, friction constant F3, cycle guard F6, step guard F24), so the critical path is tractable, but the hydrograph/runoff corrections change computed outputs and require re-baselining the suite.

## Roadmap

- **P0** — Fix the 5 critical and 6 high correctness defects that corrupt stamped results: WA storm typo (F17), C→CN inversion (F5/F22), bioretention ×3600 (F12), culvert friction constant (F3), pipe-arch radius crash (F1), BMP zero-cost ranking (F14), continuous-sim RNG overflow (F23), Clark UH cfs conversion (F7) and zero-histogram (F8), GVF supercritical sign (F2), Muskingum-Cunge timestep (F9).
  - _Each produces silently-wrong numbers an engineer stamps, or (F1) crashes an entire conduit type. Shipping any of these to the App Store risks a professional relying on incorrect hydrology/hydraulics — a liability and reputational blocker. Several are one-liners (F17, F3, F23)._
- **P0** — Add the two cheap, auto-fix-safe crash/hang guards a reviewer can trivially trigger: cycle guard in LongestPathFrom (F6, uncatchable StackOverflow tears down Civil 3D) and the elevStepFt<=0 guard in BuildStorageIndicationCurve (F24, infinite loop).
  - _An App Store reviewer feeding a malformed or edge-case model must not be able to crash or freeze the host CAD process; these are low-risk guard-only changes._
- **P0** — Lock every P0 fix with a numeric regression test against a hand-computed reference (HDS-5 HW, TR-55 CN anchors C≈0.95→CN~98, unit-volume checks for all three UH methods, a real 43×27 pipe-arch). Do not merge a correctness fix without its anchor test.
  - _The systemic cause of these bugs shipping is that tests assert only loose magnitude bounds (F31); numeric anchors are the only durable guard against regression and against the remaining transposition-class errors._
- **P1** — Resolve the medium correctness findings: WQ load double-count (F15), box wetted perimeter (F11), pond-chain tailwater double-subtraction (F13), inlet-control Hc/D (F4), HGL label key collision (F26), and the ~27-jurisdiction drawdown window (F18). Investigate the PLAUSIBLE Atlas14 column-shift (F29) and confirm/fix.
  - _These distort BMP sizing, HGL capacity, rating curves, drawing labels and compliance pass/fail. Non-conservative (F11) and false-fail (F18) cases are the priority within this tier._
- **P1** — Close the license-validation gap (F28): treat 4xx and non-JSON/non-boolean server responses as denials, reserving the offline stub for true transport failures. Add the remaining input guards (F25 Years, F27 report write) and the fetcher response-size cap (F30).
  - _F28 lets any well-formed token defeat server-side validation — a revenue/enforcement hole. F27 turns an I/O failure into an unhandled-exception dialog in front of a reviewer._
- **P1** — Expand the test suite beyond magnitude bounds across the engine (F31): numeric anchors for SCS/CN, culvert control transition, hydrograph unit-volume and inter-method consistency, and a StateComplianceData table-validation test asserting sane storm/drawdown ranges for every jurisdiction.
  - _The loose-assertion pattern is why F3, F4, F5, F17 all shipped; a table-range test alone would have caught F17 and F18._
- **P2** — Fix I/O fidelity and guidance issues: LandXML ArchPipe export (F21), DXF multi-line labels (F20), SCS flood-storm fallback (F19), and wet-pond HSG-A suitability (F16).
  - _Lower user impact (cosmetic, round-trip fidelity, or advisory), but each is a visible-quality gap for a professional tool and worth clearing before a second release._
- **P2** — Adopt structural defenses against the recurring error classes: introduce typed/dimensioned quantities (or at minimum a unit-audit checklist) to prevent SI-in-US-customary and missing-conversion bugs; add a CI cross-check that the three hydrograph methods agree within tolerance and each encloses one unit; regenerate StateComplianceData from its source generator with automated range validation; and add an adversarial malformed-input fuzz harness for the network/topology parsers.
  - _Four of the five critical defects were dimensional errors and the compliance table was a hand-transcription typo — process/tooling changes, not just point fixes, are what prevent the next batch from shipping._
